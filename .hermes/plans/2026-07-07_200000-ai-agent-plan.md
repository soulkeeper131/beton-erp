# AI Agent за Beton ERP — Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Добавяне на вграден AI чат в Beton ERP + външен API за function calling, който позволява на потребителите да управляват платформата чрез естествен език.

**Architecture:** 
- Чат UI компонент (sidebar widget) → `/api/agent/chat` → LLM (DeepSeek/GPT) с tool definitions → извиква съществуващи API-та → резултат с потвърждение
- Външни AI клиенти → `/api/tools` (function definitions като JSON Schema) → директни REST заявки с API ключ → стандартни endpoint-и
- System prompt дефинира домейна (бетонов бизнес), entities, workflows и правила за използване

**Tech Stack:** 
- DeepSeek API (за chat completions) — вече имаме ключ в Bitwarden
- React компонент за чат UI (вграден в layout)
- Vercel AI SDK или директен fetch към OpenAI-compatible endpoint
- Streaming responses за чата

## Твоята роля

## Домейн познания

### Бетонови типове и цени
Системата има каталог с типове бетон (C12/15 до C50/60), като всеки има клас на якост и цена на m³.
Цените са в лева (лв) и се определят от администратора.

### Услуги
Освен самия бетон, се предлагат услуги:
- Полагане на бетон (ръчно или с помпа)
- Полагане на замазка (с/без фибри, различна дебелина)
- Транспорт на бетон
- Помпени услуги (стационарна помпа, автопомпа 32м/42м/52м)

Всяка услуга има варианти с конкретно описание, мерна единица и цена.

### Работен процес
1. Създава се клиент → 2. Дефинира се обект (строителна площадка) → 3. Прави се оферта → 4. Извършва се бетониране → 5. Създава се акт

Офертата може да има множество редове (различни типове бетон и услуги).
Актът се свързва с оферта и проследява реално изпълнените количества.

### Правила
- **ВИНАГИ** извиквай get_catalog() преди create_offer() — за да знаеш точните продукти и цени
- **ВИНАГИ** искай потвърждение преди създаване/редакция — показвай какво точно ще се направи
- **НЕ** измисляй цени — използвай цените от каталога
- **НЕ** измисляй клиенти/обекти — търси ги първо, предлагай създаване само ако не съществуват
- За дати използвай формат YYYY-MM-DD
- За суми използвай лева (лв)
- Всички документи (оферти, актове) се генерират по Образец 19
- Фактурите са в евро (EUR)

- Бъди конкретен и кратък
- Когато показваш суми, форматирай ги с 2 знака след запетая
- Когато предлагаш действие, покажи какво ще се случи и искай потвърждение

## Фаза 2: Chat API Endpoint

### Task 3: Създай `/src/app/api/agent/chat/route.ts` — chat endpoint

Endpoint-ът приема съобщения, праща ги на LLM с tool definitions, и обработва отговора:

```
POST /api/agent/chat
Body: { messages: [{ role: "user", content: "..." }] }

Response (streaming): 
  - Ако LLM иска да извика tool → връща tool_call събитие
  - Ако LLM връща текст → stream-ва текста
  - След tool execution → връща резултата
```

Имплементация:
- Използва OpenAI-compatible API (DeepSeek)
- Поддържа streaming (SSE)
- Контекстна памет (последните N съобщения)
- Валидация на tool параметрите
- Логване в audit_log

### Task 4: Създай API ключ за LLM достъп

Вземи DeepSeek API ключ от Bitwarden и го конфигурирай в настройките.

### Task 5: Добави конфигурация на AI agent в `/settings`

- Поле за API ключ (чете се от Bitwarden)

## Отворени въпроси

1. **Кой LLM?** DeepSeek (вече имаме ключ) е най-евтин и поддържа function calling. Алтернативно GPT-4o, но е по-скъп.

2. **Колко памет?** Да пазим ли цялата история на чата в базата, или само последните 20 съобщения в паметта?

3. **Мулти-tenant?** Всеки потребител вижда само своите данни. Tool handlers трябва да филтрират по userId.

4. **Rate limiting?** За chat endpoint-а трябва rate limit (напр. 30 заявки/минута), за да не изгорим бюджета за API.
